# ATPEZMS -- Global Design Blueprint

This document is the **Level 1 Global Blueprint** required by `DESIGN_RULES.md`. It defines the system architecture, bounded contexts, cross-cutting rules, and technology decisions for ATPEZMS. All vertical slice designs must align with this blueprint, and if a slice reveals a flaw here, this document is updated first.

This document defines **what** the system is and **why** it is shaped this way. It does not define **how** things are implemented -- that belongs in `IMPLEMENTATION.md` (global standards) and per-slice design/implementation documents.

---

## 1. System Architecture

### 1.1 Architectural Style: N-Tier Layered

ATPEZMS follows a standard N-Tier layered architecture. Requests flow top-to-bottom through well-defined layers, each with a single responsibility:

```
  HTTP Request
       |
       v
+--------------+
|  Controller   |  Accepts HTTP requests, validates input shape, delegates to
|  Layer        |  service. Returns HTTP responses. Contains NO business logic.
+------+-------+
       |
       v
+--------------+
|  Service      |  All business rules live here. Orchestrates operations across
|  Layer        |  repositories and other services. Defines transaction boundaries.
+------+-------+
       |
       v
+--------------+
|  Repository   |  Data access only. Spring Data JPA interfaces that translate
|  Layer        |  method calls into SQL. No business logic.
+------+-------+
       |
       v
+--------------+
|  Database     |  H2 embedded relational database.
+--------------+
```

The primary value is **separation of concerns**. Each layer has one job. If the billing calculation changes, only the billing service changes -- the controller and repository are unaffected. If we swap the database, only the repository layer changes.

It also enforces a **dependency direction**: controllers depend on services, services depend on repositories, but never the reverse. A repository never calls a controller.

### 1.2 Data Transfer Objects

DTOs sit at the layer boundaries and serve as the contract between the controller and the outside world:

- **Request DTOs** define what the client sends. They carry validation annotations so that input validation happens at the boundary, before any business logic runs.
- **Response DTOs** define what the client receives. They decouple the API's external contract from the internal entity structure, meaning the database schema can evolve without breaking API consumers.

Entities (JPA-managed objects) never leave the service layer. Controllers receive request DTOs and return response DTOs. The service layer translates between the two.

### 1.3 Organization: Package-by-Context

The codebase is organized by **bounded context** (domain area), not by technical layer. All code for a domain area -- its controller, service, repository, entities, and DTOs -- lives under one package.

```
com.atpezms.atpezms
+-- AtpezmsApplication.java
+-- common/               # Shared infrastructure (cross-cutting)
+-- park/                  # Zones, park-wide configuration
+-- identity/              # Staff users, roles
+-- ticketing/             # Visitors, tickets, wristbands, visits
+-- billing/               # Transactions, bills
+-- telemetry/             # Scan event logging
+-- rides/                 # Ride operations, queues
+-- food/                  # Food outlets, menus, inventory
+-- merchandise/           # Stores, products, stock
+-- events/                # Shows, schedules, reservations
+-- loyalty/               # Points, tiers, benefits
+-- safety/                # Lost & found, incidents
+-- analytics/             # Reports, forecasts
```

Each context contains its own `controller/`, `service/`, `repository/`, `entity/`, `dto/` sub-packages.

Package-by-context groups related code together. Opening one package gives the complete picture of that feature. It also makes it straightforward to extract a context into its own module later. Package-by-layer (`controller/`, `service/`, `repository/` at the top level) would scatter related code across the entire source tree.

## 2. Technology Choices

| Concern | Choice | Rationale |
|---------|--------|-----------|
| Language | Java 25 | Current release. Configured by Spring Initializr. |
| Framework | Spring Boot 4.0.5 | Convention-over-configuration. Dependency injection, auto-configuration, embedded server, vast ecosystem. |
| Build | Gradle (Groovy DSL) | Generated by Spring Initializr. Dependency management, compilation, testing. |
| Web Layer | Spring MVC | Embedded Tomcat, REST controller annotations. Added via `spring-boot-starter-webmvc`. |
| Data Access | Spring Data JPA | Repository interfaces backed by Hibernate. Added via `spring-boot-starter-data-jpa`. |
| Database | H2 (embedded) | Zero-setup. File mode for persistence, in-memory mode for tests. No external DB server to manage. |
| Validation | Jakarta Bean Validation | Declarative input validation via annotations on DTOs. Added via `spring-boot-starter-validation`. |
| Schema Migration | Flyway | Versioned, repeatable SQL migrations. Hibernate DDL auto-generation is not used for schema management. |
| Testing | JUnit 5 + Spring Boot Test | Included by default. Integration and slice-level testing. |
| Security (deferred) | Spring Security | JWT-based stateless auth. Designed now, skeleton implemented after Identity slice. |

## 3. Global Infrastructure Rules

These rules apply across all bounded contexts. They are architectural constraints, not implementation details.

### 3.1 API Response Convention

API endpoint paths are grouped by bounded context under `/api/<context>/...` (e.g., `/api/ticketing/visits`, `/api/billing/transactions`). This keeps the HTTP surface aligned with the bounded-context package layout.

**Success responses** return the resource (DTO or collection) directly as the response body, with the appropriate HTTP status code. There is no wrapper envelope -- HTTP status codes already communicate success/failure, so wrapping in `{"success": true, "data": ...}` is redundant.

**Paginated collections** use Spring Data's `Page<T>` serialization format.

**Error responses** use a consistent structure produced by a centralized global exception handler. Every error response includes: HTTP status code, a machine-readable error code string, a human-readable message, a timestamp, and a field-level error list (populated for validation failures, empty otherwise).

### 3.2 Error Categories

Business errors are grouped into categories with fixed HTTP status semantics. The exception class hierarchy that implements these categories is defined in `IMPLEMENTATION.md`.

| Category | HTTP Status | When Used |
|----------|------------|-----------|
| Resource Not Found | 404 | Requested entity does not exist |
| Duplicate Resource | 409 | Creation would violate a uniqueness constraint |
| State Conflict | 409 | Operation cannot proceed given the current state of the resource (e.g., wristband already deactivated, visit already ended) |
| Business Rule Violation | 422 | Input is well-formed but violates a domain rule (capacity exceeded, insufficient stock, eligibility denied, outstanding balance, slot conflict) |
| Validation Failure | 400 | Malformed input, missing required fields, type mismatches |

**401 and 403 are reserved strictly for authentication and authorization.** No business exception maps to these codes.

### 3.3 Transactional Consistency (CO-2)

The spec requires: "All financial transactions (billing + inventory deduction) must be atomic -- completed fully or not at all."

This is achievable because ATPEZMS is a monolith with a single datasource. Operations that span multiple entities (e.g., decrement food inventory AND record a billing transaction) execute within a single database transaction. The orchestrating service method defines the transaction boundary.

Consequences:
- When context A calls context B's service within a transaction, B's work joins A's existing transaction. If either fails, everything rolls back.
- No context publishes partial state changes. Either the full operation succeeds or nothing changes.

### 3.4 Idempotency

Hardware devices (POS terminals, turnstiles, kiosks) will retry requests on network timeout. Without idempotency guarantees, retries could produce duplicate charges and inventory corruption.

**Rule:** All write operations originating from hardware devices must include a client-generated idempotency key.

**Guarantee (commit-based):** The system guarantees at-most-once *committed* processing for any given key. If a request with a previously seen key has already committed, the system returns the original response without re-executing the operation. If a request did not commit (rolled back or crashed before commit), a retry may execute again.

**Scope:** This applies to device-originated operations (POS charges, turnstile scans, kiosk reservations). Staff dashboard operations in a browser do not require idempotency keys.

### 3.5 RFID Resolution

CO-1 states the RFID wristband is the sole identifier for all visitor interactions. Every context that processes a wristband scan (Rides, Food, Merchandise, Events) receives a raw RFID tag from hardware.

**Rule:** Contexts receiving an RFID tag call Ticketing's service to resolve it to the active Visit (and thereby the Visitor, Wristband status, and entitlements). No other context performs its own RFID-to-visitor lookup. Ticketing is the single system of record for the RFID-visitor binding.

This resolution is a **hot path** (PR-1 requires < 1 second from scan to turnstile unlock). Slice designs for Ticketing and Rides must account for this performance constraint.

### 3.6 Configuration as Data

Multiple requirements depend on configurable values: daily max capacity (FR-VT3), peak/off-peak pricing (FR-VT2), ride wait-time thresholds (FR-RQ4), food reorder thresholds (FR-FB3), loyalty tier rules (FR-LY1).

**Rule:** Operational configuration is stored as database records, not application properties or hardcoded constants. Changes require Manager or Admin role and are auditable (SE-2).

**Scope:** Park-wide settings (max capacity, operating hours, seasonal calendar) are owned by the Park context. Context-specific settings (ride capacity, reorder thresholds, loyalty tier definitions) are owned by each respective context.

Application properties (`application.properties` / profiles) are reserved for infrastructure configuration only: database connection, server port, logging levels, Flyway settings.

### 3.7 PII and Encryption (SE-1, CO-3)

Visitor PII (name, contact information) and payment tokens must be encrypted at rest and in transit.

- **In transit:** All API communication uses HTTPS (CI-1). This is a deployment concern, not an application code concern.
- **At rest:** PII fields on the Visitor entity are encrypted at the application level before persistence. The specific encryption mechanism (JPA attribute converter or equivalent) is defined in the Ticketing slice design. Non-PII operational data (ride sessions, queue entries, scan events) does not require field-level encryption.
- **Payment tokens:** Tokens received from the payment gateway during checkout are encrypted before storage. Defined in the Billing slice design.

Each slice design must identify which of its fields, if any, contain PII.

### 3.8 Audit and Traceability

- **Common entity fields:** Every entity carries `id`, `createdAt`, and `updatedAt` for basic traceability. The mechanism for populating these is defined in `IMPLEMENTATION.md`.
- **Accountability fields:** Entities where change accountability matters (pricing configuration, user management, park settings) additionally carry `createdBy` and `updatedBy`.
- **Soft-delete:** The default for record deletion is hard-delete. Entities that must remain traceable use soft-delete (`deletedAt`). Specifically: lost-and-found records (SF-4), incidents (SF-3), and configuration audit records (SE-2).
- **High-impact action logging:** Administrative actions (price changes, capacity changes, user role changes, emergency overrides) are logged to an immutable audit trail. The audit mechanism is defined in `IMPLEMENTATION.md`.

### 3.9 Notification Architecture

Two distinct notification paths exist:

- **Visitor-facing notifications** (reservation confirmations, show cancellation alerts per FR-ER3): delivered via the NotificationService external interface (SMS/email, mocked during development).
- **Staff-facing alerts** (low-stock alerts per FR-FB3, ride status changes): delivered as persistent notification records stored in the database and queryable via API. Staff dashboards poll or query these. They do not go through the external NotificationService.

### 3.10 Scan Event Logging

Every wristband scan in the system (ride entry, POS charge, kiosk reservation, gate exit) generates a scan event in the Telemetry context. This data serves SE-3 (duplicate/unauthorized RFID detection) and FR-MG2 (congestion heat map).

**Rule:** The context processing a wristband scan is responsible for calling the Telemetry service to log the scan event. This is a fire-and-forget call that must not be on the PR-1 critical path. Prefer after-commit asynchronous logging; if the telemetry log fails, the primary operation still succeeds.

## 4. Security Architecture

Security implementation is deferred, but the architecture is defined now so all slices are designed compatibly. The security skeleton (JWT filter, security configuration) is implemented immediately after the Identity slice, so subsequent slices add role restrictions as they are built rather than retrofitting at the end.

### 4.1 Authentication: JWT-Based Stateless

Staff members and device service accounts authenticate by sending credentials to a login endpoint. The server validates credentials and returns a signed JWT. All subsequent requests include this token in the `Authorization: Bearer <token>` header.

The server validates the JWT on every request and does not maintain server-side sessions. The token carries the user's identity and roles. This is stateless authentication.

Why JWT over sessions: ATPEZMS is consumed by distributed hardware across the park (POS terminals, turnstiles, kiosks). Stateless tokens require no session store, any server instance can validate them, and devices can cache and reuse them.

### 4.2 Authorization: Role-Based Access Control

The JWT contains the user's granted roles. Each endpoint declares its required role(s). The server extracts roles from the token and checks them against the endpoint's requirement.

**System roles:**

| Role | Purpose | Primary Access |
|------|---------|---------------|
| `ROLE_ADMIN` | System administrators | All endpoints. User management, system configuration, security settings. |
| `ROLE_MANAGER` | Park managers | Reports, analytics, pricing configuration, all read endpoints. |
| `ROLE_TICKET_STAFF` | Ticket counter staff | Visitor registration, ticket sales, wristband issuance. |
| `ROLE_RIDE_OPERATOR` | Ride operators | Ride operations, queue monitoring, wristband scan for ride access. |
| `ROLE_FOOD_STAFF` | Food court staff | Food ordering, wristband scan for food charges. |
| `ROLE_STORE_STAFF` | Merchandise store staff | Merchandise sales, inventory management, wristband scan for store charges. |
| `ROLE_EVENT_COORDINATOR` | Event coordinators | Event scheduling, reservation management. |
| `ROLE_KIOSK` | Self-service terminals | Machine-to-machine identity for kiosk devices, authenticated via the same JWT mechanism. Kiosks are hardware, not human users, but need authenticated API access for event reservations (FR-ER1). |

### 4.3 Visitors Are Not System Users

Visitors do not log in to the system. They interact entirely through their physical RFID wristband, scanned by authenticated devices. The flow is:

1. A POS terminal (authenticated as `ROLE_FOOD_STAFF`) scans a visitor's wristband.
2. The terminal sends a request with the RFID tag in the request body.
3. The server resolves the RFID tag via Ticketing (Section 3.5), processes the operation, and charges it to the visitor's tab.

The visitor is a **data entity**, not a security principal. Their identity is their RFID wristband tag, not a username/password.

### 4.4 Security-Readiness in Pre-Security Slices

Before the security skeleton is implemented, endpoints are open. However, all slices are designed for security from the start:

- Every endpoint's slice design specifies the required role(s). Adding the enforcement annotation is a mechanical step when the security skeleton lands.
- Wristband identification is always an explicit request parameter, never inferred from a session.
- Service methods receive all data they need as parameters. They do not assume anything about the caller.

## 5. Domain Map

The system is divided into **12 bounded contexts**. Each context owns a coherent slice of the domain with clear responsibilities and boundaries.

Consolidation note: Safety and Telemetry are thin contexts. If scope pressure requires it, Safety could be folded into Park (as park-level operational concerns) and Telemetry into common infrastructure, without architectural damage.

---

### 5.1 Park

**Responsibility:** The physical and operational structure of the theme park. Zones, park-wide configuration, seasonal calendar. This is the reference data that most other contexts depend on.

**Why this context exists:** Zones are referenced by Ticketing (pass coverage), Rides (ride location), Food (outlet location), Merchandise (store location), and Events (venue location). If any single domain context owned Zone, it would create circular dependencies. Park is the shared foundation that all others depend on.

**Core entities:**
- **Zone** -- A logical area of the park (Adventure Zone, Water Zone, Food Court Area, etc.).
  Phase 1 seeds minimal Zone reference data (`code`, `name`) to unblock Ticketing entitlements (zone coverage) without building Park CRUD yet; additional descriptive/operational fields (e.g., description/status) are introduced in Phase 2 (Park CRUD slice).
- **ParkConfiguration** -- Park-wide settings: maximum daily visitor capacity, operating hours. One active configuration record at a time.
- **SeasonalPeriod** -- Date ranges marked as peak/off-peak. Referenced by Ticketing for pricing (FR-VT2) and by other contexts for scheduling.

**Relationships:** Zone is referenced (by ID) from entities in Rides, Food, Merchandise, and Events. ParkConfiguration is read by Ticketing for capacity enforcement (FR-VT3).

**FR/NFR coverage:** FR-VT2 (seasonal pricing reference), FR-VT3 (capacity configuration), SC-1 (adding zones requires no architectural changes).

---

### 5.2 Identity

**Responsibility:** Staff user accounts, credentials, and role assignments. The system's answer to "who is making this request?"

**Core entities:**
- **User** -- A staff member or device service account. Has credentials (hashed password, even before security is implemented -- the field must exist for the security slice) and one or more roles.
- **Role** -- An authority granted to a user. Maps to the roles in Section 4.2.

**Relationships:** User has many Roles (many-to-many).

**Boundary:** This context does NOT manage visitors. Visitors are data entities in Ticketing. This context also does not implement authentication mechanics (JWT generation/validation) -- that is the security slice, which builds on top of Identity's user/role data.

**FR/NFR coverage:** SE-2 (administrative function restriction via roles).

---

### 5.3 Ticketing

**Responsibility:** Visitor registration, ticket and pass sales, wristband issuance, visit lifecycle management, entitlement definition, park capacity enforcement, and RFID-to-visitor resolution (Section 3.5).

**Core entities:**
- **Visitor** -- A person visiting the park. Carries PII (name, age, height, contact information). Created at the ticket counter. Height is captured at registration for ride eligibility checks (SF-2). Persists across visits (a returning visitor is the same record).
- **Wristband** -- An RFID wristband. Has a unique RFID tag and a lifecycle status (in-stock, active, deactivated). One active wristband per visitor at a time.
- **Ticket** -- The purchase record. Links a Visitor to a PassType with a pricing snapshot (the price paid at time of purchase, which may differ from the current PassType price), validity window, and payment reference. A multi-day Ticket spans multiple Visits.
- **PassType** -- A reference/configuration entity defining pass categories (Single-day, Multi-day, Ride-specific, Family, Fast-track) and their base pricing rules.
- **Visit** -- One entry-to-exit session in the park. Created when a visitor enters (wristband activated). Ended when the visitor completes checkout (Section 5.4, Billing) and exits. Bills, transactions, and scan events all attach to a Visit, not directly to a Wristband or Visitor.
- **AccessEntitlement** -- Defines what a Ticket grants. A set of access grants of different kinds: zone access (can enter a zone), ride access (can ride a specific ride), and queue priority (fast-track). This model handles the complexity of ride-specific, family, and fast-track passes that zone-only modeling cannot.

**Key relationships:**
- Visitor has many Visits (one per park entry, potentially across days for multi-day passes)
- Visit has one Wristband
- Visit is associated with one Ticket
- Ticket references a PassType
- Ticket has many AccessEntitlements

**Cross-context dependencies:**
- Reads Zone IDs from Park (for zone-based entitlements)
- Called by every scan-processing context for RFID resolution (Section 3.5)
- Called by Billing for visit-end orchestration

**Checkout orchestration:** Ticketing orchestrates the exit sequence. When a visitor scans at the exit gate, Ticketing calls Billing to check and settle the outstanding balance. If the balance is clear, Ticketing calls the TurnstileGateway to unlock the exit gate and marks the Visit as ended. Billing owns bill aggregation and payment processing; Ticketing owns the exit workflow.

**Design assumptions:**
- Multi-day passes: each day is a separate Visit with its own wristband assignment and billing cycle. The bill is settled per-visit at exit. The post-paid exit-gate model (FR-EB2) requires settlement before exit; accumulating charges across multiple days without settlement would be impractical.
- Ride-specific passes: grant access to a named set of rides via ride-level AccessEntitlements.
- Fast-track passes: grant queue priority via a priority AccessEntitlement. The exact priority mechanism is defined in the Rides slice.
- Family passes: a pricing category for group registration. Each family member gets their own Visitor record, Wristband, and individual billing. The "family" benefit is the discounted group rate at purchase time.

**FR/NFR coverage:** FR-VT1 (pass types), FR-VT2 (pricing via PassType + SeasonalPeriod), FR-VT3 (capacity enforcement), FR-VT4 (RFID association), CO-1 (RFID as sole identifier), PR-1 (resolution performance).

---

### 5.4 Billing

**Responsibility:** The financial ledger. Records every charge to a wristband, aggregates the final bill, and processes payment at checkout.

**Core entities:**
- **Transaction** -- A single charge or credit on the ledger. Records the amount, the source context (food, merchandise, event, ticket), a description, and the Visit it belongs to. This is an **append-only ledger**: transactions are never modified or deleted after creation.
- **Bill** -- The aggregated view of all transactions for a Visit. Tracks total charges, pre-paid deposits, outstanding balance, and status (open, settled). One Bill per Visit.

**Key relationships:**
- Bill belongs to a Visit (from Ticketing, referenced by ID)
- Bill has many Transactions

**Cross-context dependencies:**
- Other contexts (Food, Merchandise, Events, Ticketing) call Billing's service to record transactions. They never write to the billing database directly.
- Ticketing calls Billing during checkout to check/settle the balance.
- Loyalty is notified when transactions are recorded (for point accrual).

**Design rule:** Billing is the single authority over all financial records. When a food order occurs, the Food service computes the charge and calls the Billing service to record it. This ensures all financial data is consistent and auditable in one place.

**FR/NFR coverage:** FR-FB1 (wristband billing -- the billing side), FR-EB1 (bill aggregation), FR-EB2 (exit control -- the balance-check side), CO-2 (transaction consistency).

---

### 5.5 Telemetry

**Responsibility:** Records every wristband scan event in the system for security monitoring and operational analytics.

**Core entities:**
- **ScanEvent** -- A log record of a wristband interaction. Captures: timestamp, RFID tag, device identifier, zone, purpose (ride-entry, POS-sale, kiosk-reservation, gate-exit, etc.), and decision (allowed/denied with reason).

**Relationships:** ScanEvent references RFID tag and zone (by ID). It does not hold JPA relationships to other contexts' entities -- it is a flat log.

**Consumers:**
- Safety/Security: SE-3 duplicate/unauthorized RFID detection operates on this log.
- Analytics: FR-MG2 congestion heat map is derived from scan event data.

**FR/NFR coverage:** SE-3 (unauthorized/duplicate RFID detection), FR-MG2 (congestion data source).

---

### 5.6 Rides

**Responsibility:** Ride operations, eligibility validation, real-time queue tracking, alternate ride suggestions, turnstile signaling, and per-session capacity enforcement.

**Core entities:**
- **Ride** -- A specific attraction. Has physical attributes (capacity per session, minimum height, minimum age), operational status (open, closed, maintenance), and belongs to a Zone (by ID, from Park).
- **QueueEntry** -- Tracks a visitor's entry into a ride's queue. Records entry time, exit time (boarded or left), enabling wait-time calculations.
- **RideSession** -- One cycle/run of a ride. Has a fixed maximum rider count (SF-1). Records which visitors boarded and when. Used for capacity enforcement and throughput analytics.

**Key relationships:**
- Ride belongs to a Zone (by zone ID, from Park)
- Ride has many QueueEntries (current queue)
- Ride has many RideSessions (history)
- RideSession has many riders (capacity-limited)

**Cross-context dependencies:**
- Calls Ticketing for RFID resolution and entitlement checking (does this visitor's pass cover this ride/zone? Do they meet age/height restrictions?). Ticketing's resolution response includes the visitor attributes (age, height) needed for eligibility checks.
- Calls Telemetry to log scan events.
- Calls TurnstileGateway to signal unlock/lock.

**Capacity enforcement (SF-1):** The Rides context enforces strict per-session capacity limits. A RideSession has a fixed maximum rider count derived from the Ride's configuration. Boarding is denied when the session is full.

**Alternate ride suggestions (FR-RQ4):** When a ride's wait time exceeds its configured threshold, the system suggests alternative rides with shorter queues. This requires a cross-ride query within the Rides context (not a cross-context call).

**FR/NFR coverage:** FR-RQ1 (eligibility validation), FR-RQ2 (turnstile actuation), FR-RQ3 (queue monitoring), FR-RQ4 (alternate suggestions), SF-1 (capacity enforcement), SF-2 (age/height validation), PR-1 (scan-to-unlock < 1s).

---

### 5.7 Food & Beverage

**Responsibility:** Food court and restaurant operations. Menu management, food inventory, order processing, low-stock alerts.

**Core entities:**
- **FoodOutlet** -- A food court, restaurant, or snack stand. Has a name, location zone (by ID from Park), and operating status.
- **MenuItem** -- An item on an outlet's menu. Has a name, price, and category.
- **FoodInventoryItem** -- Stock tracking for a menu item at a specific outlet. Tracks current quantity and the reorder threshold that triggers alerts.
- **FoodOrder** -- A record of what a visitor ordered: line items, total amount, and a reference to the Transaction created in Billing.

**Cross-context dependencies:**
- Calls Ticketing for RFID resolution.
- Calls Billing to record the transaction for each order.
- Calls Loyalty to look up the visitor's current discount tier (for FR-LY2). The discount is applied to the price before recording the Billing transaction. Must design this as an extension point -- initially a no-op until the Loyalty slice is built.
- Calls Telemetry to log the scan event.

**Staff alerts (FR-FB3):** Low-stock alerts are delivered as persistent notification records queryable via API (Section 3.9), not via the external NotificationService.

**FR/NFR coverage:** FR-FB1 (wristband billing -- the food side), FR-FB2 (inventory updates), FR-FB3 (low-stock alerts).

---

### 5.8 Merchandise

**Responsibility:** Merchandise store operations. Product catalog, stock management, cross-store availability, purchase recording.

**Core entities:**
- **Store** -- A merchandise shop. Has a name, location zone (by ID from Park), and operating status.
- **Product** -- A product in the catalog. Has a name, description, price, and category.
- **StoreInventory** -- Stock of a product at a specific store. Tracks current quantity. Enables the cross-store availability feature.
- **MerchandisePurchase** -- A record of a visitor's purchase: line items, total, and a reference to the Transaction in Billing.

**Cross-context dependencies:** Same pattern as Food -- Ticketing for RFID, Billing for charges, Loyalty for discounts (extension point), Telemetry for scan logging.

**Cross-store suggestions (FR-MS2):** When an item is out of stock at one store, the system queries inventory across all stores for that product. This is a within-context query, not a cross-context call.

**FR/NFR coverage:** FR-MS1 (inventory validation), FR-MS2 (cross-store suggestions), FR-MS3 (sales logging).

---

### 5.9 Events

**Responsibility:** Show and event management. Scheduling, seat/slot reservations, double-booking prevention, cancellation notifications.

**Core entities:**
- **Event** -- A show, performance, or special experience. Has a name, description, venue/zone (by ID from Park), and category.
- **EventSchedule** -- A specific occurrence of an event: date, time, venue, total capacity. An Event can have many scheduled occurrences.
- **Reservation** -- A visitor's booked seat/slot for a specific EventSchedule. Has a lifecycle status (held, confirmed, cancelled).

**Key relationships:**
- Event has many EventSchedules
- EventSchedule has many Reservations (capacity-limited)
- Reservation belongs to a Visit (by visit ID, from Ticketing)

**Cross-context dependencies:**
- Calls Ticketing for RFID resolution.
- Calls Billing to record a transaction if the event has a fee. For paid events, a transaction is added to the visitor's wristband tab at reservation time -- no separate payment is collected. This is consistent with the post-paid model.
- Calls NotificationService (external, mocked) for visitor-facing cancellation/reschedule alerts (FR-ER3).
- Calls Telemetry to log the scan event.

**Double-booking prevention (FR-ER2):** The system prevents double-booking by placing a short-lived hold on a slot during the confirmation window. If confirmation doesn't arrive within the hold period, the slot is released. The exact hold mechanism (database-level or application-level) is defined in the Events slice design.

**FR/NFR coverage:** FR-ER1 (reservation management), FR-ER2 (conflict resolution), FR-ER3 (notifications).

---

### 5.10 Loyalty

**Responsibility:** Loyalty program. Points accrual, tier management, benefit application (discounts), visit-history-based suggestions.

**Core entities:**
- **LoyaltyAccount** -- Tied to a Visitor (persists across visits, not a single wristband). Tracks total points and current tier.
- **LoyaltyTier** -- A reference/configuration entity defining tier thresholds and benefits (e.g., Silver at 500 points: 5% discount).
- **PointTransaction** -- A record of points earned or redeemed. Linked to a Billing Transaction by ID.

**Key relationships:**
- Visitor (by ID from Ticketing) has one LoyaltyAccount
- LoyaltyAccount has a current LoyaltyTier
- LoyaltyAccount has many PointTransactions

**Cross-context dependencies:**
- **Billing -> Loyalty (not Loyalty -> Billing):** When Billing records a transaction, it calls Loyalty to accrue points. Loyalty maintains its own point balance and tier state. Discount lookups are Loyalty-internal -- no back-call to Billing. This prevents a dependency cycle.
- Food and Merchandise call Loyalty to look up the current discount for a visitor. Loyalty returns the discount percentage/amount; the calling context applies it to the price.

**Recommendations (FR-LY3):** Provides suggestions based on visit history -- rides the visitor hasn't tried, events they might enjoy based on past attendance. This is a simple history-based heuristic, not a recommendation engine or ML model.

**Discount integration:** Food and Merchandise must design their order flows with a discount extension point from the start. Initially this is a no-op (returns zero discount). When the Loyalty slice is built, the real implementation replaces the no-op.

**FR/NFR coverage:** FR-LY1 (point tracking), FR-LY2 (benefit application), FR-LY3 (recommendations).

---

### 5.11 Safety

**Responsibility:** Lost-and-found item management, emergency incident reporting, park-wide emergency override.

**Core entities:**
- **LostItemReport** -- Filed by staff on behalf of a visitor. Describes the lost item: category, color, description, approximate date and location lost. May reference a Visitor (by ID from Ticketing) if the reporter identifies themselves.
- **FoundItem** -- Logged by staff. Describes a found item: category, color, description, date and location found, storage location.
- **Incident** -- An emergency event. Records timestamp (to the second per SF-3), location, description, severity, medical team dispatch status, and resolution.

**Item matching (FR-LF1):** A search function matches FoundItem records against LostItemReport records by category, color, and date proximity. This is a within-context query.

**Emergency override (FR-EH2):** The system accepts a command that unlocks all turnstiles park-wide for evacuation. This endpoint calls TurnstileGateway for every registered gate and logs an Incident. Accessible only to Admin and Manager roles.

**Soft-delete:** LostItemReport, FoundItem, and Incident records use soft-delete for traceability (SF-4, SF-3).

**FR/NFR coverage:** FR-LF1 (item matching), FR-EH1 (incident reporting), FR-EH2 (emergency override), SF-3 (incident logging detail), SF-4 (lost-and-found traceability).

---

### 5.12 Analytics

**Responsibility:** Reporting and forecasting. Read-only views across the system. Owns no persistent entities -- it queries data from other contexts.

**Reports:**
- **Visitor Forecasts (FR-MG1):** Daily visitor load predictions for the upcoming week based on historical visit counts from Ticketing. Uses a simple heuristic (historical averages with seasonal adjustment), not a machine learning model.
- **Congestion Heat Map (FR-MG2):** Visitor density per zone derived from ScanEvent data in Telemetry.
- **Revenue Breakdown (FR-MG3):** Financial reports segmented by source (tickets, food, merchandise). Derived from Billing Transactions.
- **Inventory Reports (FR-MG4):** Stock consumption data from Food and Merchandise inventory records.
- **Loyalty Analytics (FR-MG5):** Program engagement metrics from Loyalty data.

**Cross-context access pattern:** Analytics defines read-only repository interfaces over other contexts' tables. These repositories are co-located in the Analytics package and may only perform read operations. This is the one context where direct repository access across context boundaries is permitted, because routing every analytics query through service layers would add complexity without value (analytics never writes, so there are no business rules to bypass).

**Report export (SI-3):** Analytics owns CSV and PDF export. Generation uses in-process libraries (not an external service).

**FR/NFR coverage:** FR-MG1 through FR-MG5, SI-3, PR-3 (report generation < 10 seconds).

## 6. Cross-Context Interaction Rules

### 6.1 Service-Layer Calls, Not Repository Access

When Context A needs data from Context B, A's service calls B's service. A never accesses B's repository directly.

**Exception:** Analytics may define read-only repository interfaces over other contexts' tables (Section 5.12).

### 6.2 ID-Based References at Context Boundaries

When an entity in one context references an entity in another, it stores the **foreign key ID**, not a JPA object relationship. This prevents Hibernate from creating implicit joins across boundaries and keeps contexts loosely coupled.

Example: A FoodOrder stores a `visitId` (Long), not a `@ManyToOne Visit` field. If the Food service needs Visit details, it calls the Ticketing service.

**Within a single context**, entities use full JPA relationships (`@ManyToOne`, `@OneToMany`, etc.) freely. The loose-coupling rule applies only at context boundaries.

### 6.3 No Circular Dependencies

If Context A depends on Context B, B must not depend on A. If a circular need arises, it indicates the boundaries need re-examination or the shared concept should be extracted.

**Current dependency graph (all arrows are one-way):**

```
Park  <--  Ticketing  <--  Rides
  ^            ^              |
  |            |    +---------+
  |            |    v
  |       +-- Billing  <--  Food
  |       |    ^              |
  |       |    |    +---------+
  |       |    v
  |       +-- Loyalty  <--  Merchandise
  |                           |
  +--  Events                 |
  +--  Safety                 |
                              v
       Telemetry  <--  (all scan-processing contexts)
       Analytics  -->  (reads from all, depended on by none)
```

Specific direction clarifications:
- Billing -> Loyalty (Billing notifies Loyalty of new transactions for point accrual)
- Food/Merchandise -> Loyalty (queries discount tier)
- Food/Merchandise -> Billing (records transactions)
- Ticketing orchestrates checkout by calling Billing (check/settle balance)
- Rides/Food/Merchandise/Events -> Ticketing (RFID resolution)
- Rides/Food/Merchandise/Events -> Telemetry (scan logging, fire-and-forget)

### 6.4 Per-Slice Cross-Context Documentation

Every per-slice design document must enumerate: which other contexts it calls, for what purpose, and in which direction. This ensures the dependency graph remains explicit and cycle-free as slices are built.

## 7. External Interface Contracts

The spec defines external hardware and software interfaces. All are mocked during development.

### 7.1 Hardware Interfaces

| Interface | Spec Ref | Design Treatment |
|-----------|----------|-----------------|
| RFID Readers (HI-1) | Read wristband IDs from >= 5cm | Not mocked at the API level. The API receives the RFID tag as a string parameter. It does not matter whether a physical reader or a test script produced it. |
| Turnstiles (HI-2) | Open/close/status signals at rides and gates | Defined as a TurnstileGateway interface. Request-response semantics: the caller sends an unlock/lock command and receives an acknowledgment or timeout. A configurable timeout defines how long the system waits for acknowledgment before treating the command as failed. The mock implementation logs the action and returns immediate success. |
| POS Terminals (HI-3) | Receipt printers at food courts and stores | Not mocked at the API level. POS terminals are API clients. Receipt printing is client-side. |

### 7.2 Software Interfaces

| Interface | Spec Ref | Design Treatment |
|-----------|----------|-----------------|
| Payment Gateway (SI-1) | External card payment at checkout | Defined as a PaymentGateway interface. Accepts an amount and payment method token. Returns success/failure. The mock always returns success. |
| Notification Service (SI-2) | SMS and email for visitors | Defined as a NotificationService interface. Methods for SMS and email delivery. The mock logs the notification. |
| Report Export (SI-3) | CSV and PDF generation | Not an external service. Implemented in-process within the Analytics context using libraries. |

Each external interface is defined as a Java interface with a mock implementation as the default Spring bean. When a real integration is needed, a new implementation replaces the mock via dependency injection -- no other code changes.

## 8. Implementation Roadmap

Each slice follows the Level 2 Mini-Waterfall from `DESIGN_RULES.md`: detailed design -> implementation -> verification -> education.

| Phase | Slice | Key Deliverables | Depends On |
|-------|-------|-----------------|------------|
| 1 | **Ticketing** | Visitor, Wristband, Visit, Ticket, PassType, AccessEntitlement. Registration, wristband issuance, RFID resolution, capacity enforcement, exit orchestration. | None (Foundational) |
| 2 | **Park** | Zone, ParkConfiguration, SeasonalPeriod. Reference data CRUD. | Phase 1 |
| 3 | **Identity + Security Skeleton** | User, Role. Staff CRUD. JWT authentication filter and security configuration. All subsequent slices add role restrictions as they go. | Phase 1 |
| 4 | **Billing** | Transaction, Bill. Record-transaction API, bill aggregation, payment processing (via mocked gateway). | Phase 1 (visits) |
| 5 | **Telemetry** | ScanEvent. Logging service, duplicate detection queries. | Phase 2 (zones) |
| 6 | **Rides** | Ride, QueueEntry, RideSession. Eligibility check, turnstile signaling, queue tracking, alternate suggestions. | Phase 1 (RFID/entitlements), Phase 5 (scan logging) |
| 7 | **Food** | FoodOutlet, MenuItem, FoodInventoryItem, FoodOrder. Order flow, inventory, low-stock alerts. Discount extension point (no-op until Loyalty). | Phase 1 (RFID), Phase 4 (billing) |
| 8 | **Merchandise** | Store, Product, StoreInventory, MerchandisePurchase. Purchase flow, cross-store lookup. Discount extension point. | Phase 1 (RFID), Phase 4 (billing) |
| 9 | **Events** | Event, EventSchedule, Reservation. Reservation flow, hold mechanism, cancellation notifications. | Phase 1 (RFID), Phase 4 (billing, optional) |
| 10 | **Safety** | LostItemReport, FoundItem, Incident. Item matching, incident logging, emergency override. | Phase 1 |
| 11 | **Loyalty** | LoyaltyAccount, LoyaltyTier, PointTransaction. Point accrual, discount lookup, recommendations. Replaces no-op discount hooks in Food/Merchandise. | Phase 4 (billing), Phase 7-8 (discount hooks) |
| 12 | **Analytics** | Report endpoints, visitor forecasts, congestion map, revenue breakdown, inventory reports, loyalty metrics. CSV/PDF export. | All prior phases (read access) |

**Dependency rationale & YAGNI Principle:**
- We do not have a separate "Infrastructure" phase. Dependencies (Spring Data JPA, Validation, Flyway, etc.) and global elements (Exception Handlers, Base Entities) are introduced *Just-In-Time* (YAGNI) as the first vertical slice demands them.
- Ticketing (1) is the core foundational slice. It creates visitors, wristbands, visits, and the RFID resolution service that all scan-processing contexts depend on. It also introduces the initial Spring Boot skeleton and required database dependencies organically.
- Park (2) provides zones and config that most contexts reference.
- Identity + Security (3) establishes user/role infrastructure and authentication early, avoiding a retrofit.
- Billing (4) is the ledger that Food, Merchandise, and Events write to.
- Telemetry (5) is in place before Rides, so scan logging works from the first ride scan.
- Rides (6) is the highest-priority operational feature.
- Food (7) and Merchandise (8) follow a similar pattern and both depend on Billing.
- Events (9) is relatively self-contained after the core infrastructure exists.
- Safety (10) is mostly independent; could be built earlier if prioritized.
- Loyalty (11) depends on Billing transaction data and replaces discount hooks in Food/Merchandise.
- Analytics (12) reads from everything, so it goes last.
